# Threat Model: TMI (Threat Modeling Improved) Server

## 1. System context

TMI is the Go back-end for a collaborative security-review SaaS. It exposes a REST API (OpenAPI 3 over Gin) and a WebSocket protocol (AsyncAPI over Gorilla) for real-time data-flow-diagram editing, persisted via GORM to PostgreSQL (development) or Oracle ADB (production), with Redis for cache and session state. Tenants are security teams who store **threat models** — structured analyses of their own systems' weaknesses — so a confidentiality breach has amplified downstream impact: leaking a customer's threat model hands an attacker a map of that customer's soft spots.

Identity is federated: TMI is an OAuth 2.0 / OIDC / SAML relying party against upstream IdPs (Google, GitHub, Microsoft, custom) and mints its own JWTs (HS256/RS256/ES256). RBAC is enforced per resource (reader/writer/owner) by middleware on the request path. The server is also an HTTP **client** on behalf of users: it fetches user-supplied document URLs and parses DOCX/XLSX/PPTX/PDF/HTML, pulls files from linked Google Drive / Microsoft 365 / Confluence accounts via stored OAuth tokens, delivers HMAC-signed webhooks to user-registered endpoints, and feeds an LLM embeddings/RAG pipeline ("Timmy"). It ships as a hardened Chainguard distroless container (nonroot, read-only filesystem, all capabilities dropped) deployable via Terraform + Kubernetes to OCI, AWS, GCP, Azure, or Heroku. Roughly 707 Go source files; OpenTelemetry tracing throughout. The companion front-end (tmi-ux, Angular) is out of scope here except where server output becomes its input.

**Tenancy and trust model.** TMI is deployed **single-tenant per instance**: every user of a given deployment belongs to the same organization. There is no `tenant_id` column anywhere in the schema. The intra-tenant confidentiality boundary is the `is_confidential` flag on threat models and survey responses, enforced via the built-in **Confidential Project Reviewers** group (UUID `00000000-…-0003`); non-confidential resources are organizationally readable by design. Workflow integrity is enforced by the **Security Reviewers** role (who may change threat-model `status` and `threat` objects) and the **automation** account flag (machine clients via client_credentials).

**Design principle: server is the sole authn/authz enforcer.** All access-control decisions are made server-side. The Angular client (tmi-ux) implements guards, resolvers, and `canEdit` checks for UX only; they are expected to be bypassable and are not part of the security boundary. Any authorization gap is therefore unambiguously a server bug, and any server-side fix is sufficient — no client change is required to close an authz finding.

## 2. Assets

| asset | description | sensitivity |
|---|---|---|
| Tenant threat models & diagrams | Customers' security analyses: identified threats, assets, DFDs, mitigations — effectively documentation of the customers' own weaknesses | critical |
| Authorization state (ACLs) | Per-resource reader/writer/owner grants binding users and groups to threat models | critical |
| JWT signing keys | HS256 secret or RS/ES256 private key used to mint every TMI access token | critical |
| SAML SP private key & cert | Service-Provider key for signing AuthnRequests and decrypting IdP assertions | critical |
| Upstream IdP client secrets | OAuth client_secret values registered with Google/GitHub/M365/Confluence/etc. | critical |
| Stored third-party tokens | Per-user upstream-IdP and content-provider OAuth access/refresh tokens (Drive, M365, Confluence) | critical |
| DB & Redis credentials | DSN/passwords for PostgreSQL/Oracle and Redis | critical |
| Issued refresh tokens & client_credentials | Long-lived bearers TMI issues to browsers and machine clients | high |
| User PII & group memberships | Emails, display names, IdP claims, group/role assignments driving ACL decisions | high |
| Survey templates & confidential responses | Security-intake questionnaires + answers, with an `IsConfidential` flag; may contain compliance/vuln detail | high |
| Webhook URLs & HMAC secrets | User-registered callback URLs and per-subscription signing secrets | high |
| Audit trail | Append-only change log + version snapshots used for accountability and rollback | high |
| Collaborative session state | Active WS sessions, participant lists, presenter role, deny lists | medium |
| System settings | Admin-tunable runtime config: rate limits, feature flags, provider configs, encryption-key references | medium |
| Service availability | Liveness of API + WS hub; outage halts collaborative editing and intake workflows | medium |

## 5. Deprioritized

| threat | reason |
|---|---|
| Memory-corruption RCE in the TMI server process | Go is memory-safe; `CGO_ENABLED=0`; no `unsafe` in request-handling paths. Native-code risk is confined to dependencies and covered under T4 (supply chain). |
| Spoofing of authenticated REST requests | JWT signature validation in middleware is the control; failure modes are covered under T1 (identity confusion) and T6 (signing-key disclosure) rather than as a separate spoofing row. |
| CSRF on state-changing REST endpoints | API is bearer-token authenticated via `Authorization` header, not cookie-session by default, so classic CSRF does not apply. Revisit if HttpOnly-cookie auth becomes the primary client path. |
| Repudiation on public endpoints | No state-changing actions reachable unauthenticated; nothing to repudiate. |
| Tampering of JWKS / OpenID-configuration responses | Public read-only metadata derived from server keys; tampering requires server compromise, covered by T4/T6/T11. |
| XXE in SAML / OOXML XML parsing | Go's `encoding/xml` does not resolve external entities by default; bounded decoder further restricts. Treated as a sub-case of T12 should a non-default decoder be introduced. |
| Client-side localStorage token theft (#164) | Client-side concern (tmi-ux), out of scope for this server model; T17 (stored XSS) is the relevant server-side contribution. |
| Cross-user read of **non-**`is_confidential` resources within the single tenant | Per owner trust model: all users of an instance belong to one organization; non-confidential threat models are organizationally readable by design. Such a read is an organizational-hygiene bug, not a confidentiality breach. The confidentiality boundary is the `is_confidential` flag (covered by T5). |
| Unauthenticated Grafana/Jaeger/Prometheus in `docker-compose.prod.yml` | Owner states this compose stack is never internet-reachable; "prod" in the filename is a misnomer. The OTLP-span and log-leak components remain in T23. |

## 6. Open questions

- **Addon authority model (T18).** Owner does not know whether addon write-backs via client_credentials act with the addon-owner's authority or the invoking user's. Affects T18 likelihood and status. Verify by: trace a real addon invocation through `api/addon_*` and the client_credentials token's `sub` claim to see whose ACL is checked on the write path.
- **Timmy retrieval filter (T13).** `timmy_embeddings` carries `threat_model_id` and session listing filters on it, but the similarity/retrieval query was not located (#214 backend incomplete). Affects T13 likelihood (cross-user-leak component). Verify when the retrieval handler lands: confirm the vector-search SQL includes `WHERE threat_model_id = ?`.
- **`ResolveUser` fuzzy-lookup exploitability (T1).** The `(provider, email)` fallback at first-login resolution: can an attacker whose IdP returns a victim's email (unverified-email IdP, or email re-use after account deletion) bind their first login to the victim's `InternalUUID`? Affects T1 likelihood. Verify by: code-walk `auth/handlers_oauth_user.go` sparse-record completion against an unverified-email IdP profile.
- **Server-side WS auth path (T20, T23).** Server does not read `?token=` from the WS URL; tmi-ux still appends it (AUTH-001). Confirm what the server actually authenticates on (Authorization header via middleware? subprotocol? cookie?) and file a tmi-ux bug to stop appending the token. Affects T20 controls, T23 likelihood. Verify by: read `api/websocket.go` upgrade path and the route's middleware chain.
- **OTLP span redaction (T23).** `RedactSensitiveInfo` covers slogging; no equivalent found on OTLP span attributes before export. Affects T23 status. Verify by: grep for span `.SetAttributes` / OTLP exporter config and check whether request bodies / Authorization headers are attached.
- **T3 downgrade condition.** Owner disputes T3 severity.
