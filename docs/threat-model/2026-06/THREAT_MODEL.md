# Threat Model: TMI (Threat Modeling Improved)

## 1. System context

TMI is a collaborative threat-modeling platform: a Go REST API server (~963 Go/TS source files, github.com/ericfitz/tmi, Gin + OpenAPI 3) with real-time collaborative diagram editing over WebSocket (Gorilla, AsyncAPI), an Angular frontend in a separate repo (tmi-ux), PostgreSQL (dev) or Oracle ADB (prod) persistence via GORM, and Redis for caching and session state. Authentication is federated — OAuth 2.0/OIDC and SAML (Google, GitHub, Microsoft, custom IdPs) — with the server minting its own JWTs (HS256/RS256/ES256). Deployments are single-tenant (Kubernetes, Heroku, or local Docker); the in-tree Terraform is dated and its threat model is explicitly deferred until the deployment story settles.

The architecture is mid-refactor in a security-relevant way: document extraction and embedding have moved out of the monolith onto sandboxed worker instances (`cmd/extractor`, `cmd/chunkembed`) in a Kubernetes cluster, orchestrated by a component controller over NATS JetStream (job subjects + Object Store for blobs). The monolith fetches all external content itself and remains the sole DB writer; workers receive blob references only — no URLs, no credentials, no DB access — and the extractor runs with `egress: none`. This worker boundary is the bulk of the new threat surface since the project's original threat model (`docs/THREAT_MODEL.md`, 2026-04-30). A second structural shift: most configuration migrated out of files/env vars into the database (encrypted at rest via `internal/crypto/settings_encryptor.go`), administered through the `/admin/*` API.

Other security-relevant surfaces: an LLM/RAG assistant ("Timmy") over extracted content, delegated OAuth integrations into customer content stores (Google Drive, MS365, Confluence), HMAC-signed webhooks, an addon ecosystem with scoped delegation tokens, RBAC (reader/writer/owner) with an `is_confidential` intra-tenant boundary, and an append-only audit trail enforced by DB-level triggers with an operator-pinned out-of-band alert sink. The project is security-mature — gitleaks/CodeQL/Dependabot/SBOM in CI, design-doc-driven security work (`docs/superpowers/specs/`), and a steady stream of landed fixes from its own prior threat-model response plan. Its users are security teams; the data it holds is itself a map of customer organizations' weaknesses.

## 5. Deprioritized

| threat | reason |
|---|---|
| Hostile insider with legitimate admin role subverting the platform | Risk accepted by owner: "the admin could just modify the running code or run additional code on the deployed instance using the instance's credentials — there are a hundred rabbit holes, and at the end of the day the admin is the one person you have no choice but to trust." Controls against *hijacked* admin sessions remain in scope (T1). |
| Permissive Terraform/deployment-template defaults (was draft T5: gcp-public Cloud SQL public IP, dev Redis unauthenticated, provisioning-window public k8s API) | Deferred by owner: the Terraform and deployment options are dated; a focused threat model will be performed when the deployment story settles. Includes terraform-state aspects of secrets handling. |
| Repudiation of user actions on the REST API | DB-trigger-enforced append-only audit trail with version snapshots and admin audit log covers attribution |
| Stored XSS via diagram cell content / threat descriptions | Risk-accepted per design: server stores raw text; sanitization is the Angular client's responsibility (tmi-ux #164, outside this codebase) |
| Tenant data exfiltration via webhook payloads to subscriber-controlled URLs | By-design integration channel; HMAC signing proves origin and subscribers are admin-configured (prior-model T22, risk-accepted) |
| Spoofed webhook deliveries at subscriber endpoints | HMAC-SHA256 signatures with per-subscription secrets authenticate origin; subscriber-side verification is outside this system's boundary |
| Information disclosure via verbose error responses | Low-impact stack-detail leakage; structured error helpers exist and an error-response audit is tracked |
| Tampering with NATS extraction messages in transit | Bus is cluster-internal; an attacker positioned to inject NATS traffic is covered by T15/T19 |
| DoS against static file serving | Bounded by the platform load balancer and trivial static reads; no amplification available |

## 6. Open questions

- [Owner-states] The extractor pod's `egress: none` posture is enforced by NetworkPolicy rendered from the TMIComponent CR. Affects: T19 controls/likelihood. Verify by: tracing the controller's NetworkPolicy rendering and adding a conformance test that a deployed extractor cannot resolve DNS or reach the internet.
- [Owner-states] Workers never touch the database ("monolith is the sole DB writer"). Largely code-verified (internal/worker has no DB imports; result consumer persists), but worth a CI guard (lint/import check) so the invariant survives future contributors. Affects: T19 status.
- [Owner-states] No validation is performed on result blobs before persistence. Affects: T19 controls. Closing this is a section 8 row.
- The reserved `source_url`/`source_secret_ref`/`fetch_limits` envelope mode is unused by design (#347). The invariant "extractor never holds a fetch credential" is currently enforced by non-use, not by schema or code. Tripwire: before any future code-extractor enables it, require the shared egress-guard library and a fetch-controlled egress class on the CR.
- Per-component NATS authentication/authorization (accounts/creds scoping subjects and Object Store buckets per worker) is left to deployment configuration; the specs don't mandate it. Decide whether it becomes a design-level requirement while the feature is unfinished.
- WebSocket heartbeat JWT re-validation and per-user connection caps: no landed commits found; status unverified. Affects: T17.
- OTLP span redaction is unverified — traces may carry bearer material the log redaction already strips. Affects: T2 (token-leakage channel).
- Is SAML actually deployed anywhere? crewjam/saml is a sensitive dependency to carry unused. Affects: T2 surface breadth.
- The live dev OAuth client secrets in local `config-development.yml`: not committed (verified), but consider rotation hygiene and per-developer OAuth apps so a lost laptop doesn't burn shared credentials. Affects: T3.
- Deferred: focused threat model for the deployment/Terraform story when it settles (see section 5).
